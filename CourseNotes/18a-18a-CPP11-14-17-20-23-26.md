# The Evolution of C++: 11 → 14 → 17 → 20 → 23 → 26

Your textbook has chapters framed around C++20/23/26. This course is built and
compiled against **C++17**. This lecture explains what actually changed across
these revisions, which of those changes matter for the OOP fundamentals this
course teaches, and — for the newer standards — what the textbook's syntax
*means* even though we won't use it here.

C++ gets a new ISO standard roughly every 3 years. Each one is a superset of
the last (old code keeps compiling); the question for any given course or
project is just how much of the *new* surface area is worth adopting.

---

## C++11 — the big one

C++11 is the standard that turned C into "modern C++." If you only ever learn
one revision beyond the original language, this is the one, and it's the one
this course leans on most heavily.

- **`auto`** — type deduced from the initializer.
- **Range-based `for`** — `for (auto& x : container)` instead of manual iterators.
- **`nullptr`** — a real, typed null pointer constant, replacing `NULL`/`0`.
- **Smart pointers** (`std::unique_ptr`, `std::shared_ptr`) — RAII for
  dynamically allocated memory. This is exactly the technique from
  [Dynamic Memory & RAII](./03f-DynamicMemory&RAII.md) and is used all over
  `Files/Pages2` (`std::vector<std::unique_ptr<Shape>>`).
- **`override` and `final`** — the compiler checks that a virtual function
  actually overrides a base class function, catching typo'd signatures. Directly
  relevant to [Abstract Classes & Interfaces](./04b-AbstractClass.md).
- **Move semantics / rvalue references (`&&`)** — lets objects transfer
  ownership of resources instead of copying them. This is *why* the "double
  free on copy" problem from the practice exam even has a modern fix
  (move instead of copy).
- **`= delete` / `= default`** — explicitly disable or request compiler-generated
  constructors/operators (e.g. "this class cannot be copied").
- **Lambdas** — anonymous inline functions, `[capture](params) { body }`.
- **`static_assert`** — compile-time assertions.

Everything the midterm/practice exam asks about (constructors, RAII, abstract
classes, the Rule of Three) is either pre-C++11 language or *solved cleanly* by
a C++11 feature. That's not a coincidence — this is where the language's OOP
story became complete enough to teach from.

---

## C++14 — a cleanup release

Small, mostly quality-of-life fixes on top of C++11:

- Generic lambdas (`auto` in lambda parameters).
- `auto` return type deduction for regular functions.
- Relaxed rules for `constexpr` functions (loops/branches allowed).
- Binary literals (`0b1010`) and digit separators (`1'000'000`).

Nothing here changes how you'd teach or use OOP. If a course "skips" a standard,
this is the one to skip.

---

## C++17 — where this course's toolchain sits

C++17 is a real, useful step up, and it's what our Makefiles compile with
(`-std=c++17`).

- **Structured bindings** — `auto [x, y] = somePair;` unpacks a pair/tuple
  directly into named variables.
- **`if constexpr`** — compile-time branching in templates, relevant once you
  reach [Class Templates](./15b-ClassTemplates.md).
- **Class template argument deduction (CTAD)** — `std::pair p(1, 2.0);`
  instead of `std::pair<int, double> p(1, 2.0);`.
- **`std::optional`, `std::variant`, `std::any`** — safer alternatives to
  sentinel values / raw unions / `void*`.
- **Nested namespaces** — `namespace A::B::C { ... }` instead of nesting braces.
- **`std::filesystem`** — a standard library for file/path operations.

This is the standard where the *library* got more expressive, but the *object
model* (classes, inheritance, virtual functions) didn't change at all from
C++11. That's exactly why C++17 is a good place for this course to sit: modern
enough to write clean code, without introducing language machinery that isn't
about OOP.

## C++20 — where things start diverging from this course's scope

C++20 is a large release, and this is roughly where your textbook's chapters
start assuming features we don't use:

- **Concepts** — named, compile-time constraints on template parameters
  (`template <std::integral T> ...`). Conceptually, this is like an **interface
  for template parameters** — it plays a similar role to the abstract classes
  from [04b-AbstractClass.md](./04b-AbstractClass.md), but for generic code
  instead of runtime polymorphism.
- **Ranges** — a `<ranges>` library that lets you write pipelines over
  containers (`std::views::filter`, `std::views::transform`) instead of raw
  iterator loops from [Vectors & Iterators](./04c-Vectors&Iterators.md).
- **Modules** — a replacement for `#include`-based header files. Solves the
  compile-time and macro-leakage problems `#pragma once`/`#ifndef` were
  invented for (see [05c-ifndef&pragma.md](./05c-ifndef&pragma.md)), but
  build-system support is still uneven, which is part of why we don't use it.
- **Coroutines** — `co_await`/`co_yield`, for writing asynchronous code that
  reads like synchronous code.
- **The spaceship operator `<=>`** — one operator that generates `<`, `>`,
  `<=`, `>=` automatically, relevant once you've seen
  [Overloading Operators](./13a-OverloadingOperators.md).
- **Designated initializers** — `Point p{.x = 1, .y = 2};`, C-style struct
  initialization by field name.

None of this changes *how a class is designed*. It changes how you write
generic/template code, how you structure large builds, and how you write
async code — all things this course intentionally doesn't get into.

## C++23 — refinement, not revolution

- **`std::expected<T, E>`** — a return type for "a `T` on success, or an `E`
  error" without exceptions — same problem [Exceptions](./09d-Exceptions.md)
  solves, different mechanism.
- **`std::mdspan`** — a non-owning view over multi-dimensional data (a cleaner
  way to think about the 2D array indexing from the midterm's array questions).
  Multidimensional `operator[]` (`m[i, j]`) — one call, multiple indices.
- **`deducing this`** — lets a member function deduce whether it's called on
  an lvalue/rvalue/const object, mostly a library-author feature.
- More `<ranges>`/`<algorithm>` additions.

Like C++14, this is a consolidation release relative to C++20.

---

## C++26 — where the language is heading

Still being finalized as compilers catch up, but the direction is set:

- **Contracts** (`pre`/`post`/`contract_assert`) — a step toward the
  encapsulation invariant-checking from
  [Open/Closed Principle](./04e-OpenClosedPrinciple.md)/
  [SOLID Principles](./04f-SOLIDPrinciples.md) being expressible *in the
  language itself*, rather than by convention/documentation.
- **Static reflection** — code that can inspect a type's own structure (its
  members, names) at compile time. This is what eventually replaces a lot of
  hand-written serialization code, relevant to
  [JSON Parsing](./09c-JSONParsing.md).
- **`std::execution` (senders/receivers)** — a standard vocabulary for async
  and parallel work, successor-ish to coroutines' plumbing.

These are genuinely useful, and also genuinely not what a first course in
object-oriented design needs to cover.

## Why this course stays on C++17

| Standard | What it added                          | Relevant to OOP fundamentals? |
|----------|-----------------------------------------|--------------------------------|
| C++11    | smart pointers, move semantics, `override`, lambdas | **Yes — this is the toolkit the course teaches with** |
| C++14    | minor cleanups                          | No |
| C++17    | structured bindings, `optional`/`variant`, CTAD | Mild convenience, not core |
| C++20    | concepts, ranges, modules, coroutines    | No — generic/library/async features |
| C++23    | `expected`, `mdspan`, more ranges        | No — same category as C++20 |
| C++26    | contracts, reflection, `std::execution`  | No — and not yet finalized |

The reasoning, plainly: **encapsulation, inheritance, polymorphism, RAII, and
the design patterns this course teaches were all fully expressible by C++11**,
and C++17 gives you a couple of nice ergonomic upgrades (structured bindings,
`optional`) without adding anything that changes how a class or a hierarchy is
designed. Everything C++20 and later adds is aimed at template/generic
programming, build systems, or concurrency — real, important areas, but ones
that belong in a *later* course, not a sophomore-level intro to OOP.

If you read ahead in the textbook and see `template <std::integral T>` or
`std::views::filter(...)` or `m[i, j]`, this lecture is the map back to "what
category of feature is this, and which lecture's concept does it rhyme with" —
you're not missing anything from *this* course by not writing it yourself.

## Quick reference: textbook syntax → course concept

| If you see this in the textbook...        | ...it plays a similar role to |
|--------------------------------------------|-------------------------------|
| `template <std::integral T>` (concepts)    | Interfaces / abstract classes ([04b](./04b-AbstractClass.md)), applied to templates |
| `std::views::filter`/`transform` (ranges)  | Iterating a `std::vector` ([04c](./04c-Vectors&Iterators.md)), but chainable |
| `auto [x, y] = pair;` (structured bindings)| Unpacking a `std::pair`/`std::tuple` |
| `a <=> b` (spaceship operator)             | Overloading `<`, `>`, `==` ([13a](./13a-OverloadingOperators.md)) all at once |
| `std::expected<T, E>`                      | An alternative to `try`/`catch` ([09d](./09d-Exceptions.md)) |
| `import module;`                           | An alternative to `#include`/`#pragma once` ([05c](./05c-ifndef&pragma.md)) |
| `contract_assert(...)`                     | Enforcing a class invariant, made a language feature instead of a convention |
